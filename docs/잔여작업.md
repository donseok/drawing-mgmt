## 2. 카드 상세

### F4-01 / F4-02 — Approval/ApprovalStep status enum 정리

**현 상태:** schema에 `Approval.status: { PENDING | IN_PROGRESS | APPROVED | REJECTED | CANCELLED }`, `ApprovalStep.status: { WAITING | APPROVED | REJECTED }`. R3c-1 BE는 contract(4종 / 3종)에 맞추기 위해 `IN_PROGRESS → PENDING`, `WAITING → PENDING`으로 응답 시점에 collapse. 코드 어디서도 `IN_PROGRESS`를 의미 있게 활용 안 함.

**작업:**
- (a) **schema 단순화**: `IN_PROGRESS` 제거, `WAITING → PENDING`으로 enum 정리. 마이그레이션 + 기존 row backfill (`UPDATE ... SET status='PENDING' WHERE status IN ('IN_PROGRESS', 'WAITING')`).
- (b) **응답 매핑 함수 제거**: `apps/web/app/api/v1/objects/[id]/approval/route.ts`의 collapse 로직 제거.
- (c) **FE 합성 로직 제거**: `apps/web/app/(main)/objects/[id]/page.tsx`에서 "current entry의 첫 PENDING step → IN_PROGRESS" 합성 부분.

**연결 코드:**
- `apps/web/prisma/schema.prisma` (Approval, ApprovalStep, ApprovalStatus, StepStatus)
- `apps/web/app/api/v1/objects/[id]/approval/route.ts`
- `apps/web/app/(main)/objects/[id]/page.tsx` (활동/결재 adapter)

**현 상태:** R3c-3에서 `Promise.allSettled`로 다중 삭제 발사. 부분 실패 시 toast 한 줄("삭제에 실패했습니다.")만 노출 → 어느 행이 실패했는지 사용자가 모름.

**작업:**

- 실패 사유별 그룹핑 (권한 / 상태 / 네트워크 등).

**연결 코드:**
- `apps/web/app/(main)/search/page.tsx` `handleBulkDelete`

### F4-04 — `User.deletedAt` 신설

**현 상태:** `prisma/schema.prisma`의 `User` 모델에 `deletedAt` 컬럼 없음. 사용자 폐기는 `employmentType=RETIRED` flip으로만 표현. soft-delete 의미가 모호.

**작업:**
- `User.deletedAt: DateTime?` 추가 + 마이그레이션.
- `requireUser`/`/api/v1/users/search` 등에서 `deletedAt IS NULL` 필터 추가.
- 기존 RETIRED 운영을 어떻게 정리할지 결정 (deletedAt + employmentType 둘 다 유지 vs RETIRED를 deletedAt으로 마이그레이션).

**연결 코드:**
- `apps/web/prisma/schema.prisma` (User)
- `apps/web/app/api/v1/users/search/route.ts`, `apps/web/lib/auth-helpers.ts`
