## 2. 카드 상세

### F4-01 / F4-02 — Approval/ApprovalStep status enum 정리

**현 상태:** schema에 `Approval.status: { PENDING | IN_PROGRESS | APPROVED | REJECTED | CANCELLED }`, `ApprovalStep.status: { WAITING | APPROVED | REJECTED }`. R3c-1 BE는 contract(4종 / 3종)에 맞추기 위해 `IN_PROGRESS → PENDING`, `WAITING → PENDING`으로 응답 시점에 collapse. 코드 어디서도 `IN_PROGRESS`를 의미 있게 활용 안 함.

**작업:**
- (a) **schema 단순화**: `IN_PROGRESS` 제거, `WAITING → PENDING`으로 enum 정리. 마이그레이션 + 기존 row backfill (`UPDATE ... SET status='PENDING' WHERE status IN ('IN_PROGRESS', 'WAITING')`).
- (b) **응답 매핑 함수 제거**: `apps/web/app/api/v1/objects/[id]/approval/route.ts`의 collapse 로직 제거.
- (c) **FE 합성 로직 제거**: `apps/web/app/(main)/objects/[id]/page.tsx`에서 "current entry의 첫 PENDING step → IN_PROGRESS" 합성 부분.

**연결 코드:**
- `apps/web/prisma/schema.prisma` (Approval, ApprovalStep, ApprovalStatus, StepStatus)
- `apps/web/app/api/v1/objects/[id]/approval/route.ts`
- `apps/web/app/(main)/objects/[id]/page.tsx` (활동/결재 adapter)

### R22 상세 (첨부 마스터 변경 + 삭제)

R21 업로드의 자연스러운 후속 — 첨부 CRUD 마무리.

- **BE `PATCH /api/v1/attachments/:id`** (신설): `{ isMaster: boolean }` 단일 필드. 트랜잭션 내 다른 master demote 후 promote (1 master per Version 보장). 단독 master를 demote하려는 시도는 `E_STATE_CONFLICT`로 거절(Version master-less 상태 회피). state-machine 가드 = R21 업로드와 동일. ActivityLog `OBJECT_ATTACH` (promote metadata).
- **BE `DELETE /api/v1/attachments/:id`** (신설): row 삭제 + storage 디렉토리 best-effort cleanup. 마스터 첨부는 sibling이 있으면 거절(다른 첨부를 master로 먼저 지정해야 삭제 가능). sibling 없는 단독 master는 안전 삭제(Version 빈 채로 남음 — 다음 업로드 시 재사용). ActivityLog `OBJECT_DETACH`.
- **FE InfoTab**: 각 첨부 row에 ⋯ 메뉴 (마스터 지정 / 삭제). 마스터 지정은 이미 master면 disabled. 삭제는 ConfirmDialog로 가드(R3c-3 패턴). edit gate 동일.
- **검증**: typecheck pass, build pass. 신규 라우트 `/api/v1/attachments/[id]` (PATCH+DELETE) 등록.

### R21 상세 (자료 첨부 업로드 — multipart)

기존 dev/upload 외에 사용자-facing 첨부 추가 흐름이 없던 것을 자료 상세에서 직접 가능하게.

- **BE `POST /api/v1/objects/:id/attachments`** (신설, multipart): 200MB cap (env `ATTACHMENT_MAX_BYTES`). state-machine 가드 (NEW / CHECKED_IN / CHECKED_OUT-by-locker만), EDIT 권한 검증. SHA-256 checksum + storagePath = `<attachmentId>/source.<ext>` (기존 `/api/v1/attachments/[id]/file` reader 호환). 트랜잭션 내 Revision/Version lazy create + Attachment row + isMaster 자동/명시 처리(다른 master demote). meta.json 사이드카로 filename/mime 보존. ActivityLog `OBJECT_ATTACH`.
- **FE `<AttachmentUploadDialog />`** (신설): 단일 파일 picker + isMaster 체크박스 + size preview. 첫 첨부면 isMaster pre-check. 성공 시 `objects.detail` + `objects.activity` invalidate. `apiRequest` 직접 호출(FormData 처리).
- **InfoTab**: state가 NEW/CHECKED_IN/CHECKED_OUT일 때만 "첨부 추가" 버튼 노출 (BE state machine과 동일 가드).

### R11 상세 (DWG 뷰어 Phase 3 — INSERT/블록 인서트)

자체 DWG 뷰어가 실용적 도면을 처리할 수 있도록 BLOCKS 섹션 + INSERT 엔티티 지원.

- **파서 dependency 순서 변경**: 기존 sequential walk → 두 단계 (1) 모든 SECTION 범위를 prescan, (2) HEADER → TABLES → BLOCKS → ENTITIES 순서로 처리. 일부 DXF writer가 ENTITIES를 BLOCKS 전에 emit하는 경우에도 INSERT가 BLOCKS 사전참조 가능.
- **BLOCKS 처리** (`readBlocks` / `readOneBlock`): BLOCK 정의를 `BlockDef { name, basePoint, entities[], inserts[] }`로 메모리 dict에 저장. 블록 내부의 LINE/CIRCLE/ARC/POLYLINE/LWPOLYLINE/TEXT/MTEXT는 그대로 entities[]에. 블록 내부의 INSERT는 nested inserts[]로 분리 (재귀 expand 위해).
- **INSERT 처리**: ENTITIES 섹션의 INSERT는 파서에서 즉시 expand (transform 적용). 블록 미존재 시 `unsupported`에 `INSERT(blockName)`으로 기록. 결과는 평탄화된 일반 entity로 출력 → scene builder는 변경 없음.
- **affine transform**: `translate(-base) → scale(sx, sy) → rotate(θ) → translate(insertion)`. Non-uniform scale은 ARC/CIRCLE에서 ellipse가 되지만 평균 스케일로 근사 (Phase 3 한계).
- **nested INSERT**: `composeInserts`로 outer transform이 inner insertion에 적용된 새 InsertHeader 합성. `MAX_NESTED_BLOCK_DEPTH=6` cycle 가드.
- **layer/color 상속**: 블록 내부 entity의 layer가 `'0'`이거나 color가 `ByLayer`/`ByBlock`일 때 INSERT의 layer/color로 대체 — DXF 표준 동작.
- **검증**: typecheck pass, build pass.
